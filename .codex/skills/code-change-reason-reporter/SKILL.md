# Code Change Reason Reporter

在實作與驗證完成後，直接在對話視窗列出「檔案位置 + 狀態 + 原因」清單，讓使用者可以快速理解每個改動的目的，而不是只看到 diff。

## 工作流程

1. 先確認本回合的程式碼改動已完成，包含必要驗證與整理。
2. 整理「本回合實際動到的檔案」清單；不要直接照抄整個 `git status`，避免把使用者原本就存在的髒變更混進來。
3. 視需要用 `git diff --name-status`、`git diff -U0`、`git diff -- <file>` 輔助定位新增、修改、刪除的位置。
4. 依語意拆分說明項目：
   - 整個檔案新增或刪除：寫絕對檔案路徑。
   - 既有檔案的局部新增、修改、刪除：寫 `絕對檔案路徑:行號`。
   - 同一檔案若有多個不同目的的改動，拆成多條，不要硬併成一個籠統理由。
5. 每條都要寫出「為什麼要改」，聚焦需求、資料流、型別、商業規則、重構目的或 bug 修正；不要只是把程式碼換句話說。

## 撰寫規則

- 狀態只使用：`新增`、`修改`、`刪除`。
- 原因要描述改動目的，不要寫成「新增一個 import」「刪掉一段 JSON」這種只有表面動作的句子。
- 若某個刪除是因為被真實 API、store、共用元件或新流程取代，要明確寫出取代來源。
- 若某個新增是為了支援型別、顯示邏輯、狀態同步或資料映射，要點出該責任。
- 無法確定原因時，回頭根據需求、任務紀錄與實作脈絡補足，不要猜測。
- 這份說明只涵蓋本回合實際處理的內容；不要替未改動的檔案補理由。
- 不要把這份說明另外輸出成文件、Markdown 檔或其他 artifact；只需要在對話視窗內直接顯示。

## 輸出格式

使用固定格式，逐條列出：

```md
/absolute/path/to/file
狀態：刪除
原因：替換成真實 API 資料，移除舊有的假資料

/absolute/path/to/file.vue:11
狀態：新增
原因：引入商品標記顯示資料型別 `ProductSelectionProductBadge`
```

## 回覆時機

- 在最終回覆中，將這份內容放在獨立的「改動原因」段落，直接顯示在對話視窗內即可。
- 若本次任務沒有任何程式碼改動，不要硬湊這個段落。
- 若使用者另外要求 code review、摘要或驗證結果，仍先遵守原本任務要求，再補上這份原因清單。
